# /ingest — 외부 기술/패턴을 Aidy 프로젝트에 흡수

외부 URL(블로그, 문서, 레포, 영상)을 분석하여 Aidy 멀티에이전트 시스템에 적용 가능한 부분을 추출하고, 적용까지 한다.

## 호출 방식

```
/ingest <URL>                    # URL 분석 → 적용
/ingest <URL> dry                # 분석만, 적용 안 함
/ingest <키워드>                 # WebSearch로 관련 자료 탐색 후 분석
```

---

## Phase 1: 소스 확보 + 교차 검증

1. URL이면 WebFetch로 본문 확보
2. 키워드면 WebSearch로 상위 3개 소스 확보
3. **교차 검증**: 최소 2개 독립 소스에서 동일하게 언급되는 것만 "검증된 패턴"으로 간주
4. 1개 소스에만 나오는 주장은 "미검증"으로 표시

## Phase 2: Aidy 적용 가능성 분류

발견한 패턴을 4 카테고리로 분류:

| 카테고리 | 적용 대상 | 예시 |
|---------|----------|------|
| **harness** | 관제 시스템 개선 | hooks, gate 강화, 안전장치 |
| **compound** | 지식 축적 개선 | 문서화, 회고, ADR 자동화 |
| **server** | 백엔드 기술 | Spring Boot 패턴, DB, API |
| **client** | iOS/Android 기술 | TCA, Compose, 네트워킹 |

## Phase 3: 적용 계획

각 패턴에 대해:
1. **현재 Aidy에 있는가?** (이미 있으면 스킵)
2. **적용 난이도** (즉시 / 다음 WO / ADR 필요)
3. **토큰 ROI** — 적용했을 때 토큰 절약 또는 품질 향상 효과
4. 구체적 파일 경로 + 변경 내용

## Phase 4: 즉시 적용 (dry가 아닌 경우)

난이도 "즉시"인 것들을 바로 적용:
- specs/ 수정 → api-contract.md, conventions.md
- gates/ 수정 → checklist, security
- .claude/ 수정 → settings.json, commands, agents
- CLAUDE.md 수정
- 워커에게 변경사항 전파 필요 시 inbox에 알림

## Phase 5: 보고

```markdown
# /ingest 결과: <소스 제목>

## 발견한 패턴 (N건)
| 패턴 | 카테고리 | 적용 | 상태 |
|------|---------|------|------|

## 즉시 적용 (N건)
- [파일경로] 변경 내용

## 다음 WO (N건)
- 패턴 설명 → WO 발행 필요

## 미검증 (단일 소스)
- 패턴 설명 — 추가 검증 필요

## 출처
- 원본: [제목](URL)
- 교차: [소스](URL)
```

## 안티패턴

- ❌ 단일 소스만 보고 적용 — 교차 검증 필수
- ❌ "좋아 보여서" 적용 — 토큰 ROI 계산 먼저
- ❌ 기존 패턴과 충돌하는 것 적용 — ADR 먼저
- ❌ 워커에게 알리지 않고 스펙 변경
