# ADR-004: 메모리 추출 Multi-Agent Pipeline 검토

**상태**: 검토 중
**날짜**: 2026-04-16
**관련**: P-006, WO-001 후속

## 배경

현재 Aidy의 메모리 추출은 단일 LLM 호출로 처리:
```
사용자 메시지 → Claude 1회 호출 → reply + memoriesExtracted
```

ai-study wiki의 Multi-Agent Pipeline 패턴(moneyflow 13 에이전트)에서 영감을 받아,
메모리 추출 품질을 높이기 위한 멀티 에이전트 구조를 검토한다.

## 현재 구조 (단일 에이전트)

```
[사용자 메시지]
    ↓
[Claude Haiku] → reply + memoriesExtracted
    ↓
[Layer 1-4 검증] (WO-005)
    ↓
[저장]
```

**장점**: 단순, 빠름, 저비용
**단점**: 추출 누락 (복합 정보), 카테고리 오분류, 맥락 부족

## 제안: 2단 파이프라인

단일 호출을 2단으로 분리:

```
[사용자 메시지]
    ↓
[Agent 1: Responder] → reply (대화 응답만)
    ↓ (reply + 원본 메시지)
[Agent 2: Extractor] → memoriesExtracted (추출만)
    ↓
[Layer 1-4 검증]
    ↓
[저장]
```

### Agent 역할 분리

| Agent | 모델 | 역할 | 온도 |
|-------|------|------|------|
| Responder | Haiku 4.5 | 대화 응답 생성 | 0.7 |
| Extractor | Haiku 4.5 | 구조화된 메모리 추출 | 0.2 |

### 왜 분리하는가

1. **전문화**: 응답 생성과 정보 추출은 다른 능력. 한 프롬프트에 두 가지를 요구하면 둘 다 품질 저하.
2. **프롬프트 최적화**: Extractor는 JSON 출력에 최적화된 system prompt 사용 가능.
3. **독립 개선**: 응답 품질과 추출 정확도를 독립적으로 튜닝 가능.
4. **비용 중립**: 같은 모델(Haiku)이라 토큰 비용 차이 미미. 레이턴시 증가는 ~200ms.

## 대안 검토

### A. 현행 유지 (단일 호출)
- 장점: 단순, 레이턴시 최소
- 단점: 추출 품질 한계
- **판정**: WO-005 (5-Layer 검증)로 품질 보완 가능. 당장은 이것으로 충분할 수 있음.

### B. 2단 파이프라인 (제안)
- 장점: 전문화, 독립 개선
- 단점: 레이턴시 +200ms, 복잡도 증가
- **판정**: 추출 정확도가 실측으로 문제될 때 도입.

### C. 3단 파이프라인 (Responder + Extractor + Judge)
- 장점: LLM-as-Judge로 품질 채점
- 단점: 비용 3배, 레이턴시 +500ms
- **판정**: v0.3 이후 검토. 현재는 과도.

## 결정

**현재: A (현행 유지) + WO-005로 검증 레이어 강화**

이유:
1. 아직 추출 품질 실측 데이터가 없음. 문제가 없는데 복잡도를 올릴 필요 없다.
2. WO-005의 5-Layer 검증이 단일 호출의 약점을 상당 부분 보완.
3. Layer 5 (품질 로깅)로 추출 정확도 데이터를 쌓은 후 판단.

**전환 트리거**: Layer 5 로그에서 추출 실패율 > 15%이면 B안으로 전환.

## For AI Agents

- 현재는 단일 호출 유지. 파이프라인 분리하지 마.
- WO-005 완료 후 Layer 5 로그를 모니터링하여 전환 판단 근거를 축적.
- 전환 시 Responder/Extractor를 별도 Service로 분리하되, 기존 AiClient wrapper 재사용.
